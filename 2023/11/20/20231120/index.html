认识微服务架构

发表于2023-11-20|更新于2023-11-26|后端
|阅读量:

微服务简介

  • 一组小的服务
  • 独立的进程
  • 轻量级通信
  • 基于业务能力
  • 独立部署
  • 无集中式管理

Loosely coupled service oriented architecture with bounded context

基于有界上下文的，松散耦合的⾯向服务的架构

微服务的利弊

优势

  • 强模块化边界
  • 可独立部署
  • 技术多样性

劣势

  • 分布式系统复杂性
  • 最终一致性
  • 运维复杂性
  • 测试复杂性

康威法则

设计系统的组织，其产生的架构设计等价于组织间的沟通结构

什么时候开始引入微服务

单块服务优先，业务规模复杂、团队规模扩展后（例如团队规模达到百人）引入微服务

什么样的组织架构更适合微服务

从传统职能型到跨智能产品型

微服务中台战略

大中台、小前台，强化技术中台和业务中台，前端业务更小更灵活，赋能业务的持续创新，快速响应市场需求。

服务分层

  • 基础服务： 核心领域服务、公共服务、中间层服务（Middle Tier Service）
  • 聚合服务：适配服务、Backend for Frontend（BFF）、边界服务（Edge Service）
  • 统称微服务或SOA

微服务总体架构体系

  • 接入层：负载均衡器；运维团队维护

  • 网关层：反向路由，线路熔断、安全等跨横切面的功能

  • 业务服务层：包括基础服务和聚合服务

  • 支撑服务层：注册发现、集中配置、容错限流、认证授权、日志聚合、监控告警、后台服务（MQ、缓存、Job等）

  • 平台服务层：开始引入新的发布体系，比如容器技术，Kubernetes等；包括发布系统、集群资源调度、镜像治理、资源治理、IAM

  • 基础设施层：计算、网络、存储、监控、安全、IDC（数据中心）的管理；运维团队维护

服务发现

独立LB

LB用硬件的F5负载均衡器或者软件的ngnix负载均衡器

当服务提供方上线以后，向运维申请域名，运维配置负载均衡器Load Balancer，将域名指向到后台服务；

服务消费方Consumer访问服务，通过DNS先做域名解析，域名解析到LB（Load Balancer）上，LB根据域名负载均衡（Load Balancing）调用（Invoke）到后台服务Service Provider

最简单；消费者成本比较低；服务的配置、域名的配置需要运维介入；集中式LB（可能是个单点）挂掉会影响整个服务无法访问；消费者调用服务必须穿透LB，可能会产生一部分性能开销

进程内LB

LB移到了应用进程内，服务提供方Service Provider 自动注册到服务注册表Service Registy，定期发送心跳Registy & keep alive，服务消费者consumer客户端带有服务发现和负载均衡功能，LB调用后台服务，定期同步服务注册表中信息

没有集中式LB，性能会更好；多语言环境中，需要为每一个消费者开发客户端，升级和多语言支持成本较高。

主机独立进程LB

LB以独立进程的方式独立部署在独立主机上

服务提供方Service Provider 自动注册到服务注册表Service Registy，定期发送心跳Registy & keep alive，LB定期同步服务注册表中信息，消费者要调用服务时，调用的是主机的LB，LB做负载均衡和远程调用

既没有单点和性能问题，也可以支持多语言，可以灵活接入；每台主机上部署LB进程，运维成本较高

微服务网关

API服务网关原理

API服务网关相当于门卫，屏蔽内部微服务细节，对外提供统一接口

用户接入设备User Interface –> 负载均衡器Load balancer –> 网关层API Gateway –> 微服务Micro-Services

网关可以部署多个，提高系统的稳定性和可用性

网关功能：

  • 反向路由，将外面的请求转换程内部的服务
  • 认证安全，对请求进行身份认证验证，阻止爬虫和黑客行为等
  • 限流熔断，流量洪峰时承担限流熔断职责，防止内部服务瘫痪
  • 日志监控，对所有流量进行访问审计，保存并分析日志，进行流量监控

开源网关Zuul

Zuul 核心是 Servlet，运行在tomcat容器中；核心组件 ZuulFilter Runner（Zuul执行filter的处理器），管理了 Zuul 内部所有的过滤器 Filter；过滤器分为3个层次，前置路由过滤器 Pre routing filters、路由过滤器 routing filters、后置路由过滤器 Post routing filters，过滤器都是Groovy脚本 可以动态插拔

请求进来先到 Servlet，Servlet将请求传递给ZuulFilter Runner，先经过前置路由过滤（比如日志处理，路由功能），再经过路由过滤器（找到目标服务进行调用），再经过后置路由过滤器（事后处理，日志统计审计等）

过滤器上传加载机制：Filter publisher 过滤器开发编译完成后上传到过滤器存储数据库中 Filter Persistence，Filter Poller 轮询Persister中的filter并将新filter推送至网关的过滤器目录Filter Directory，Filter File Manager 网关的过滤器文件管理器会定期的扫描目录，如果有新的过滤器，通过Filter Loader 加载到网关的Filter Runner，过滤器生效。

Request Context （Zuul Request的上下文）可以在各个阶段的过滤器分享相关信息

Request先经过前置路由过滤，也可以加一些自己定制的过滤器 Customer Filters，再经过路由过滤器，调用后台微服务Origin Server，再经过后置路由过滤器，如果出错，执行Error Filters，最后以Response的形式回到客户端。

微服务服务路由发现体系

支撑服务：服务注册中心（Netflix开源组件Eureka）和网关层（Netflix开源组件Zuul）

内部微服务：基础服务层、聚合服务层

内部服务之前发现：通过服务发现中心，基础服务注册到服务发现中心，聚合服务调用时基础服务时，通过服务注册中心发现，拉取路由表，实现调用；内部服务调用，直连客户端负载

外部LB客户端调用服务：通过网关层，网关层同步对应通过自己要放出去的服务列表，网关通过路由表实现调用聚合服务

服务治理：服务注册中心可以对服务的调用进行安全管控

微服务配置中心

作用：

  • 没有配置中心，配置放在配置文件中，配置不标准，格式不统一，上线后修改，周期比较长，一旦有重大故障，需要该配置或重新发布，需要的时间长对业务影响大，配置中心可以实时修改配置，比较灵活
  • 如果没有完整的配置系统，对配置的修改没有审计功能，后期无法追溯。

可以作为配置的东西：连接字符串（数据库连接字符串、缓存连接字符串、消息队列连接字符串）、可以动态调整的参数（客户端超时设置、限流的阈值）、业务的开关（比如某个功能只在某个时间生效）

服务连到配置中心以后，可以实时更新配置，更新方式有两种：服务主动到服务中心拉取配置pull，服务中心推送配置给服务push。两种方式各有利弊，pull能保证一定能拉取到配置，push能实时更新，但可能由于网络问题无法实时推送成功

常见的配置中心：spring cloud config、百度discount、携程Apollo

Apollo高可用性的设计：

  • 缓存机制:Apollo带有客户端（支持Java或.net客户端），配置中心更新以后，首先会拉取到客户端缓存中，为了防止缓存丢失，Apollo客户端能定期将配置缓存到本地。即使是配置中心挂了或者客户端重启了，因为本地缓存文件还在，还是能将配置更新到应用程序中，因此具有高可用性。

  • 两种配置更新方式:Apollo配置中心将配置更新推送到Apollo客户端、Apollo客户端定时拉取Apollo配置中心配置，保证最终能把更新的配置同步到客户端。

携程apo开源项目：http://github.com/ctripcoup/apollo

微服务通信方式RPC vs REST

RPC REST
耦合性 强耦合 松散耦合
消息协议 二进制thrift,protobuf,avro 文本XML,JSON
通信协议 TCP HTTP/HTTP2
性能 高 一般低于RPC
接口契约IDL thrift，protobuf IDL swagger
客户端 强类型客户端，一般自动生成多语言 一般HTTP客户端可访问，可自动生成强类型客户端，支持多语言
案例 阿里Dubbo、腾讯Tars、新浪motan、谷歌grpc、thrift spring MVC/spring Boot，Jax-rs，dropwizard
开发者友好 客户端比较方便，但二进制消息不可读 文本消息开发者可读，浏览器就可以访问
对外开放 对外一般需要转换成REST/文本协议 直接可以对外开放
