认识微服务架构

发表于2023-11-20|更新于2024-01-19|后端
|阅读量:

微服务简介

  • 一组小的服务
  • 独立的进程
  • 轻量级通信
  • 基于业务能力
  • 独立部署
  • 无集中式管理

Loosely coupled service oriented architecture with bounded context

基于有界上下文的，松散耦合的⾯向服务的架构

微服务的利弊

优势

  • 强模块化边界
  • 可独立部署
  • 技术多样性

劣势

  • 分布式系统复杂性
  • 最终一致性
  • 运维复杂性
  • 测试复杂性

康威法则

设计系统的组织，其产生的架构设计等价于组织间的沟通结构

什么时候开始引入微服务

单块服务优先，业务规模复杂、团队规模扩展后（例如团队规模达到百人）引入微服务

什么样的组织架构更适合微服务

从传统职能型到跨智能产品型

微服务中台战略

大中台、小前台，强化技术中台和业务中台，前端业务更小更灵活，赋能业务的持续创新，快速响应市场需求。

服务分层

  • 基础服务： 核心领域服务、公共服务、中间层服务（Middle Tier Service）
  • 聚合服务：适配服务、Backend for Frontend（BFF）、边界服务（Edge Service）
  • 统称微服务或SOA

微服务总体架构体系

  • 接入层：负载均衡器；运维团队维护

  • 网关层：反向路由，线路熔断、安全等跨横切面的功能

  • 业务服务层：包括基础服务和聚合服务

  • 支撑服务层：注册发现、集中配置、容错限流、认证授权、日志聚合、监控告警、后台服务（MQ、缓存、Job等）

  • 平台服务层：开始引入新的发布体系，比如容器技术，Kubernetes等；包括发布系统、集群资源调度、镜像治理、资源治理、IAM

  • 基础设施层：计算、网络、存储、监控、安全、IDC（数据中心）的管理；运维团队维护

服务发现

独立LB

LB用硬件的F5负载均衡器或者软件的ngnix负载均衡器

当服务提供方上线以后，向运维申请域名，运维配置负载均衡器Load Balancer，将域名指向到后台服务；

服务消费方Consumer访问服务，通过DNS先做域名解析，域名解析到LB（Load Balancer）上，LB根据域名负载均衡（Load Balancing）调用（Invoke）到后台服务Service Provider

最简单；消费者成本比较低；服务的配置、域名的配置需要运维介入；集中式LB（可能是个单点）挂掉会影响整个服务无法访问；消费者调用服务必须穿透LB，可能会产生一部分性能开销

进程内LB

LB移到了应用进程内，服务提供方Service Provider 自动注册到服务注册表Service Registy，定期发送心跳Registy & keep alive，服务消费者consumer客户端带有服务发现和负载均衡功能，LB调用后台服务，定期同步服务注册表中信息

没有集中式LB，性能会更好；多语言环境中，需要为每一个消费者开发客户端，升级和多语言支持成本较高。

主机独立进程LB

LB以独立进程的方式独立部署在独立主机上

服务提供方Service Provider 自动注册到服务注册表Service Registy，定期发送心跳Registy & keep alive，LB定期同步服务注册表中信息，消费者要调用服务时，调用的是主机的LB，LB做负载均衡和远程调用

既没有单点和性能问题，也可以支持多语言，可以灵活接入；每台主机上部署LB进程，运维成本较高

微服务网关

API服务网关原理

API服务网关相当于门卫，屏蔽内部微服务细节，对外提供统一接口

用户接入设备User Interface –> 负载均衡器Load balancer –> 网关层API Gateway –> 微服务Micro-Services

网关可以部署多个，提高系统的稳定性和可用性

网关功能：

  • 反向路由，将外面的请求转换成内部的服务
  • 认证安全，对请求进行身份认证验证，阻止爬虫和黑客行为等
  • 限流熔断，流量洪峰时承担限流熔断职责，防止内部服务瘫痪
  • 日志监控，对所有流量进行访问审计，保存并分析日志，进行流量监控

开源网关Zuul

Zuul 核心是 Servlet，运行在tomcat容器中；核心组件 ZuulFilter Runner（Zuul执行filter的处理器），管理了 Zuul 内部所有的过滤器 Filter；过滤器分为3个层次，前置路由过滤器 Pre routing filters、路由过滤器 routing filters、后置路由过滤器 Post routing filters，过滤器都是Groovy脚本 可以动态插拔

请求进来先到 Servlet，Servlet将请求传递给ZuulFilter Runner，先经过前置路由过滤（比如日志处理，路由功能），再经过路由过滤器（找到目标服务进行调用），再经过后置路由过滤器（事后处理，日志统计审计等）

过滤器上传加载机制：Filter publisher 过滤器开发编译完成后上传到过滤器存储数据库中 Filter Persistence，Filter Poller 轮询Persister中的filter并将新filter推送至网关的过滤器目录Filter Directory，Filter File Manager 网关的过滤器文件管理器会定期的扫描目录，如果有新的过滤器，通过Filter Loader 加载到网关的Filter Runner，过滤器生效。

Request Context （Zuul Request的上下文）可以在各个阶段的过滤器分享相关信息

Request先经过前置路由过滤，也可以加一些自己定制的过滤器 Customer Filters，再经过路由过滤器，调用后台微服务Origin Server，再经过后置路由过滤器，如果出错，执行Error Filters，最后以Response的形式回到客户端。

微服务服务路由发现体系

支撑服务：服务注册中心（Netflix开源组件Eureka）和网关层（Netflix开源组件Zuul）

内部微服务：基础服务层、聚合服务层

内部服务之前发现：通过服务发现中心，基础服务注册到服务发现中心，聚合服务调用时基础服务时，通过服务注册中心发现，拉取路由表，实现调用；内部服务调用，直连客户端负载

外部LB客户端调用服务：通过网关层，网关层同步对应通过自己要放出去的服务列表，网关通过路由表实现调用聚合服务

服务治理：服务注册中心可以对服务的调用进行安全管控

微服务配置中心

作用：

  • 没有配置中心，配置放在配置文件中，配置不标准，格式不统一，上线后修改，周期比较长，一旦有重大故障，需要该配置或重新发布，需要的时间长对业务影响大，配置中心可以实时修改配置，比较灵活
  • 如果没有完整的配置系统，对配置的修改没有审计功能，后期无法追溯。

可以作为配置的东西：连接字符串（数据库连接字符串、缓存连接字符串、消息队列连接字符串）、可以动态调整的参数（客户端超时设置、限流的阈值）、业务的开关（比如某个功能只在某个时间生效）

服务连到配置中心以后，可以实时更新配置，更新方式有两种：服务主动到服务中心拉取配置pull，服务中心推送配置给服务push。两种方式各有利弊，pull能保证一定能拉取到配置，push能实时更新，但可能由于网络问题无法实时推送成功

常见的配置中心：spring cloud config、百度discount、携程Apollo

Apollo高可用性的设计：

  • 缓存机制:Apollo带有客户端（支持Java或.net客户端），配置中心更新以后，首先会拉取到客户端缓存中，为了防止缓存丢失，Apollo客户端能定期将配置缓存到本地。即使是配置中心挂了或者客户端重启了，因为本地缓存文件还在，还是能将配置更新到应用程序中，因此具有高可用性。

  • 两种配置更新方式:Apollo配置中心将配置更新推送到Apollo客户端、Apollo客户端定时拉取Apollo配置中心配置，保证最终能把更新的配置同步到客户端。

携程apo开源项目：http://github.com/ctripcoup/apollo

微服务框架和治理

微服务治理需要将以下环节沉淀下来，变成框架和平台的部分，开发人员专注于业务逻辑的实现，框架环节或者平台环节由专门的框架人员或者平台人员管控。

服务注册发现

微服务中有几十甚至上百个服务，他们之间有错综复杂的依赖关系，这时候存在服务消费者要怎么发现服务生产者，这就是服务注册发现要解决的问题

服务负载均衡

为了应对大的流量，服务提供方一般是多份部署，就存在负载均衡问题

另外服务需要路由，如果需要支持灰度发布或者蓝绿发布这种机制，需要微服务治理考虑软路由能力

微服务监控

日志

对于后期排错非常关键

Metrics

调用量、延迟、服务的出错数的监控

调用链埋点

微服务有错综复杂的依赖关系，像网状结构，调用链监控可以快速定位问题

限流熔断

没有熔断措施，某个服务出现故障，可能导致整个应用瘫痪

安全&访问控制

有些服务涉及到敏感信息，需要有安全策略，比如黑名单，来限制服务的访问

RPC/REST

序列化 XML/JSON/二进制

好的微服务治理框架能同时支持XML/JSON/二进制序列化协议，根据场景灵活配置

微服务监控系统分层和监控架构

监控系统分层

基础设施监控

最底层，运维团队负责

涉及网络、交换机等等底层设备的监控，这些设备的稳定性直接影响到上层的服务应用的稳定性。

基础设施监控核心指标：网络流量、丢包、错包、连接数等。

系统层监控

设计物理机、虚拟机、操作系统OS等

监控核心指标：CPU、内存memory、网络使用情况network、磁盘利用率disk、IO带宽等等。

应用层监控

监控核心指标：监控url的访问性能、计数、延迟，服务service的性能，sql的运行情况，缓存cache可用率，响应时间，每秒查询率qps等等。

业务监控

核心指标监控：比如登录注册、下单、支付等。

运营层面

端用户体验监控

核心指标监控：性能、返回码、城市、地区、运营商、版本、系统等。

从哪些方面进行监控

日志监控、Metrics监控、调用链监控、告警系统、健康检查

监控架构

在虚拟机或者物理机上我们会注入服务，在主机或者服务的进程内装agent，负责收集机器和应用的日志和Metrics，发送到后台的监控系统；

一般服务比较大的话，收集的log和Metrics比较大，所以会加一个大的队列，一般是Kafka，队列的好处是两边可以解耦，如果后台系统维护的话，通过队列缓冲，这些日志信息不会丢掉；

后台一般常用的监控手段有日志监控ELK（三个开源软件的缩写：elasticsearch，Logstash，kibana，elasticsearch一个分布式搜索引擎，Logstash负责收集日志，kibana展示呈现日志相关的信息）；Metrics主流采用时间序列数据库Influx DB，通过Frafana来呈现和展示；

Sensu 是一个开源的基础设施和应用程序监控解决方案，它可以监控服务器、相关服务和应用程序健康状况，并通过第三方集成发送警报和通知。

调用链监控如何选型

调用链监控原理

微服务是一个复杂的分布式系统，如果没有一套好的调用链监控，当服务间依赖出现问题的时候会非常难以定位，目前在各大主流的互联网公司中，阿里有鹰眼监控系统，美团点评有CAT监控系统，都有相应配套的调用链监控工具来保证服务能得到有效的监控，出现问题能及时定位。调用链监控最早是Google提出的思路。

假设现在有一个应用程序，调用了WEB前端，WEB前端调用了service1，service1中调用了DB和service2，调用过程中还调用了后台的cache。

Trace：一次请求的链路过程，用trace id来标识；

Span：一个模块的调用过程，用span id来标识

Root Span：链路中第一个span对象，它的span id等同于trace id；后面生成的span都是child span；

为了维护好调用链之间的父子关系，Child Span会跟踪记录parent id；跨进程的时候，为了调用链之间能够连接，会把trace id和parent id都带过来；

调用链监控启动了之后，请求进入到WEB容器会生成一个span，调用service时也会生成一个span，每一次调用都会生成一个新的span，span是调用链形成的关键。

微服务的容错限流是如何工作的

分布式系统依赖关系错综复杂，前端请求转换到后端的服务调用的时候，一般一个请求会有很多的后端服务调用。比如一个请求会分解为6个后端服务调用，叫做6步删除，有的可能会更多。这些时候，如果后台服务出线不稳定或者延迟，没有好的限流熔断措施，有可能造成客户体验下降，严重的时候雪崩效应，整个网站挂掉。像阿里巴巴双十一这种场景，如果内部没有好的限流熔断措施，是无法支撑那么大的突发流量的。

基本概念

  • 熔断： 比如用电高,熔断设备保护,不至于危害扩大化
  • 隔离：计算机资源都是有限的，比如CPU、队列等都是限定的资源，如果不进行隔离，一个服务调用消耗很多的线程资源，把其他资源占用，导致其他服务无法访问。
  • 限流：大规模的流量突击网站的时候，要限流，比如单位时间内只允许10个请求通过。
  • 降级：如果系统后台无法提供足够的支撑能力，降级来保护系统不会进一步恶化

Netflix Hystrix 断路器原理

用了Hystrix之后，会把调用封装在Construct Hystrix Command里面，封装之后具有了熔断、隔离、限流、降级的功能。

调用模式：同步Synchronous、异步Asynchronous、响应式Reactive（也是一种异步）

请求调用之后会经过电路判断，如果电路打开的，不通的，直接返回短路，调用降级函数，如果没提供降级函数或者降级函数出错，抛出异常；如果电路闭合的，是通的，判断线程资源或者队列是否充足，如果已经满了，调用限流函数；如果电路闭合且资源充足，运行调用，如果调用超时，调用降级函数；如果执行成功了，获取正确的响应，返回到调用端；如果这个过程出错也是调用降级函数。

任何环节都会以(Metric形式)反馈给(Calculate Circuit Health)计算电路健康组件,最后综合判断,反馈给内部的熔断器电路判断组件来判断下次是打开电路还是直接熔断电路。

Docker容器部署技术和持续交付流水线

容器技术的好处：

  • 环境一致性，区别于传统交付技术，解决了环境支持引擎，比如Java会用jar包或者war包部署，不同的环境中可能造成依赖不存在的问题，容器技术把依赖的包全部打在容器镜像当中，不会存在依赖版本不一致或者缺失的问题

  • 镜像部署，原来依赖于系统的资源，发布系统一般是定制，Docker提供了镜像，把构件和发布隔离，标准镜像中既有操作系统，依赖的包，也有微服务应用程序。

基于镜像治理和多环境的持续交付流水线

通过git提交代码后，通过Jekins构建、单元测试、集成测试、打Docker镜像

将镜像上传到镜像治理中心（Docker Registry）

镜像先发布到测试环境，测试完，升级到UAT环境，打个标签给镜像

发布到UAT环境，测试完，升级到生产环境；生产环境的发布比较严格，采用蓝绿+灰度发布等机制发布到生产环境。

经过流水线，将整个交付规范起来，每一步都设置门槛，保证微服务质量，提升研发效率。

蓝绿、金丝雀和灰度发布

蓝绿发布

假定2个版本，老版本蓝色版本，新版本绿色版本

主要依赖前面网关、路由器Router,当我们发布新版本,路由器切换过去

灰度发布

提高微服务部署的稳定性和灵活性渐进方式切换，比如先把流量的10%切换成新版本，经过验证，再陆续切换。

金丝雀发布是灰度发布的一种，由于金丝雀对瓦斯极其敏感，因此以前旷工开矿下矿洞前，先会放一只金丝雀进去探是否有有毒气体，看金丝雀能否活下来，金丝雀发布由此得名。对于小规模、高风险的迭代来说，“金丝雀”策略更适合；而对于大规模且希望通过实践了解真实情况的部署来说，“灰度”则更为合适。

容器集群调度和基于容器的发布体系

单独的容器能力是比较有限的，容器资源调度平台或者容器云能够统一的管控容器资源。

虚拟机有基于虚拟机的云平台openStack，能够自动创建和管理虚拟机资源。

基于容器的资源调度平台有Mesos、k8s

资源调度框架Mesos架构

Mesos 推特，后来开源

把所有机器管理起来，抽象成一台大型计算机，有足够多的内存资源、CPU资源，把应用交给这个系统，自动完成调度。

基于Master-Slave结构，Slave机器能够运行容器的物理机和虚拟机，Master能够管理Slave机器；Master是高可用的，上面有ZooKeeper集群，leader来管理slave，leader出问题，Zookeeper会选举一个standby接管leader。

Slave会定期将资源情况、CPU情况、内存情况报告给Master，Master将信息报告给外围的Framework；master只管资源的调度，资源运用由Framework管理，可插拔机制，外围的框架可以来利用Mesos搭建自己的应用。

基于容器云发布体系

首先他有CMDB资产治理中心,他会管理企业内部资源使用情况,部门使用了多少资源、有多少应用。

发布平台，当应用要发布的时候会查询CMDB,有没有足够的配额，有的话会去Mesos启动实例，

如果我们是用Marathon(马拉松框架),他会提供API,通过API来调用相应的实例；

实例启动的时候,Mesos会去docker的镜像治理中心去拉取镜像；

服务起来后,微服务框架有自注册能力,自己注册到服务治理中心,注册服务；

发布完,可以做流量调度,有蓝绿部署,灰度部署等；发布平台调用服务注册中心的灰度部署的能力；

网关可以从服务注册中心发现新的服务,然后流量就可以调用服务灰度发布的能力,访问到Mesos里面的应用。

我们往往高估在短期内可以实现的目标，却会低估在长期内能够取得的成就。
